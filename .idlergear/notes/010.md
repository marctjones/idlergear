# Research: Existing MCP Environment/System Servers

## Executive Summary

**RECOMMENDATION: INTEGRATE EXISTING** - Good MCP servers exist for system monitoring and process management. Build thin IdlerGear wrappers for development-specific use cases.

## Category 1: System Information & Monitoring

### 1. MCP System Monitor (DarkPhilosophy/mcp-system-monitor)

**Status:** Community-maintained, Linux-focused
**Repository:** https://github.com/DarkPhilosophy/mcp-system-monitor

**Tools Provided (8 total):**
1. **get_system_info** - System details (hostname, OS, kernel)
2. **get_cpu_info** - CPU metrics (usage %, brand, frequency, cores, temp)
3. **get_memory_info** - RAM and swap statistics (total, used, free, %)
4. **get_disk_info** - Storage information (usage, mount points, filesystem)
5. **get_network_info** - Network interfaces (traffic, errors)
6. **get_processes** - Complete process listing
7. **get_process_by_pid** - Specific process details
8. **get_system_metrics** - Comprehensive system overview

**Output Format:**
- JSON-RPC 2.0 structure
- Both HTTP REST API and MCP protocol
- Clean, structured JSON responses

**Limitations:**
- Linux-focused (requires ps, df, ip, cat, hostname, uname)
- No Windows/macOS native support

### 2. System Information MCP Server (dknell)

**Package:** Available via LobeHub
**Features:**
- Real-time system metrics
- CPU usage, memory stats, disk info, network status
- Running processes
- Configurable via environment variables:
  - SYSINFO_CACHE_TTL
  - SYSINFO_MAX_PROCESSES
  - SYSINFO_ENABLE_TEMP
  - SYSINFO_LOG_LEVEL
  - SYSINFO_TRANSPORT
  - SYSINFO_HOST/PORT

## Category 2: Process Management

### 3. PM-MCP (patrickjm/pm-mcp)

**Status:** Well-maintained, production-ready
**Purpose:** Manage long-running development processes
**Repository:** https://github.com/patrickjm/pm-mcp

**Tools Provided (7 total):**
1. **start_process** - Launch commands with name, description, working directory
2. **terminate_process** - Stop individual process by ID
3. **terminate_all_processes** - Stop all managed processes
4. **list_processes** - Display all processes or details for specific one
5. **get_logs** - Retrieve stdout/stderr with optional line limits
6. **grep_logs** - Search logs using regex patterns
7. **send_input** - Send text input to running processes

**Key Features:**
- ✅ **Centralized process manager** - Lifecycle management
- ✅ **Log capture** - Both stdout and stderr
- ✅ **Log persistence** - Stored in ./pm-mcp-logs/
- ✅ **Log search** - Regex-based grep
- ✅ **Error detection** - Immediate failure reporting
- ✅ **Interactive processes** - Can send input to processes

**Use Cases:**
- Managing dev servers (npm run dev, flask run, etc.)
- Build processes
- Background tasks during development
- Long-running scripts

### 4. Kill Process MCP Server (Michal Szymanski)

**Status:** Cross-platform
**Technology:** Python psutil library
**Capabilities:**
- List processes (platform-agnostic)
- Terminate processes
- Works on Windows, macOS, Linux

### 5. MCP-PROCESS (jrame)

**Purpose:** Shell access via MCP
**Capabilities:**
- Execute commands
- Interact with filesystem via command line
- General shell access

## Category 3: Environment/Version Detection

### Package Version MCP Server (sammcj/mcp-package-version)

**Purpose:** Provide latest stable package versions

## Recommendation

**MIXED APPROACH: INTEGRATE + BUILD GAPS**

### Integrate Existing (High Quality)
1. **pm-mcp** - Process management is excellent, use as-is
2. **mcp-system-monitor** - System metrics are comprehensive
3. **mcp-package-version** - Package version checking covered

### Build IdlerGear-Specific Tools (Fill Gaps)

The missing piece is **development environment detection**. No existing server provides:
- Python/Node version detection
- Virtual environment detection (venv, virtualenv, conda, poetry)
- PATH analysis
- Project-specific environment setup
- **Consolidated developer environment info**

### IdlerGear Env Tools to Build

1. **idlergear env info**
   ```json
   {
     "python": {"version": "3.11.5", "executable": "/usr/bin/python3", "venv": "/project/.venv"},
     "node": {"version": "20.11.0", "executable": "/home/user/.nvm/versions/node/v20.11.0/bin/node"},
     "package_managers": {"pip": "23.3.1", "npm": "10.2.4", "poetry": "1.7.1"},
     "active_venv": ".venv",
     "path_entries": ["/project/.venv/bin", "/home/user/.nvm/versions/node/v20.11.0/bin", ...]
   }
   ```

2. **idlergear env activate**
   - Auto-detect and activate virtual environments
   - Add venv bin directories to PATH
   - Return activation commands

3. **idlergear env detect**
   - Detect project type (Python, Node, Rust, Go, etc.)
   - Find relevant version managers (nvm, pyenv, rbenv, etc.)
   - Return environment requirements

4. **idlergear env which**
   - Enhanced `which` command
   - Shows ALL matches in PATH (not just first)
   - Shows if command comes from venv vs system
   ```json
   {
     "python": [
       {"path": "/project/.venv/bin/python", "version": "3.11.5", "source": "venv"},
       {"path": "/usr/bin/python3", "version": "3.10.12", "source": "system"}
     ]
   }
   ```

### Why Build This?

**IdlerGear env fills a developer-specific gap:**
- Existing servers focus on **system monitoring** (CPU, memory, processes)
- IdlerGear needs **development environment awareness** (Python venvs, Node versions, PATH)
- This is lightweight, high-value, and missing from ecosystem

### Implementation Approach

**Use existing servers via integration:**
- Recommend pm-mcp for process management in .goosehints
- Recommend mcp-system-monitor for system metrics
- Document configuration

**Build `idlergear env` MCP server:**
- 4-6 focused tools (info, activate, detect, which)
- Python-based (leverages subprocess, shutil, os)
- ~500 lines of code (simple!)
- High value for AI assistants

**Token Savings Estimate:**
- Without: `which python && python --version && which node && node --version && env | grep PATH` = ~200 tokens
- With: `idlergear_env.info()` = ~80 tokens (60% reduction)
- Consolidated call vs 5+ separate commands
