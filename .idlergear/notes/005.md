## TOP 10 COMMAND-LINE TOOLS (By Frequency)

### 1. **git** (Universal - All 9 projects)
- `git commit`, `git log`, `git status`, `git diff`
- `git add`, `git push`, `git pull`
- **Pattern**: Heavy commit activity (50-100 commits per active project)
- **AI Usage**: Commit message generation, git workflow automation

### 2. **Python (pytest)** (6/9 projects: idlergear, llmfp, klareco, harbor, flex)
- `pytest` for running tests
- `pytest -v`, `pytest -s`, `pytest --tb=short`
- **Pattern**: Test-driven development, integration tests
- **AI Usage**: Test generation, debugging test failures

### 3. **Python Formatters (black/ruff)** (6/9 projects)
- `black` for code formatting
- `ruff` for linting
- **Pattern**: Pre-commit hooks, automated formatting
- **AI Usage**: Code style enforcement, auto-formatting after edits

### 4. **dotnet** (3/9 projects: pdfe, promptresponse, forebay)
- `dotnet build`, `dotnet test`, `dotnet run`
- `dotnet restore`, `dotnet publish`
- **Pattern**: C# build/test/run cycle
- **AI Usage**: Build error diagnosis, test execution

### 5. **pip/Python Package Management** (6/9 projects)
- `pip install`, `pip install -e .`
- `requirements.txt`, `pyproject.toml` management
- **Pattern**: Dependency management, virtual environments
- **AI Usage**: Dependency resolution, package updates

### 6. **mypy** (2/9 projects: llmfp)
- `mypy` for type checking
- **Pattern**: Static type analysis for Python
- **AI Usage**: Type hint generation, type error fixes

### 7. **flake8/isort** (1/9 projects: llmfp)
- `flake8` for linting
- `isort` for import sorting
- **Pattern**: Code quality enforcement
- **AI Usage**: Lint error fixes, import organization

### 8. **Makefile** (1/9 projects: llmfp)
- `make lint`, `make format`, `make test`
- **Pattern**: Task automation, common development tasks
- **AI Usage**: Understanding build workflows

### 9. **cargo** (1/9 projects: forebay - Rust)
- `cargo build`, `cargo test`, `cargo run`
- **Pattern**: Rust build system
- **AI Usage**: Rust compilation, dependency management

### 10. **npm/yarn/pnpm** (1/9 projects: forebay - Node)
- Package management for Node/TypeScript
- **Pattern**: JavaScript dependency management
- **AI Usage**: Package installation, build scripts

## MOST COMMON DEVELOPMENT WORKFLOWS

### 1. **Python TDD Workflow** (6/9 projects)
```bash
# Edit code
vim src/module.py

# Format
black src/ tests/
ruff check src/

# Test
pytest tests/ -v

# Commit
git add . && git commit -m "feat: ..." && git push
```

### 2. **C#/.NET Build-Test Workflow** (3/9 projects)
```bash
# Build
dotnet build

# Test
dotnet test --filter "TestName"

# Run
dotnet run --project ProjectName

# Commit
git add . && git commit -m "fix: ..." && git push
```

### 3. **Documentation-First Development**
- **ALL 9/9 projects** have extensive markdown docs
- CLAUDE.md, README.md updated frequently (top 5 modified files)
- **Pattern**: Document before code, AI assistant instructions

### 4. **Automation Scripts** (Heavy use across all projects)
- `run.sh`, `test.sh`, `setup.sh`, `validate.sh`
- **Pattern**: Scriptable, repeatable workflows
- **AI Usage**: Script generation, workflow automation

### 5. **IdlerGear Integration** (6/9 projects)
- `.claude/rules/idlergear.md`
- `.claude/commands/start.md`
- **Pattern**: AI knowledge management via IdlerGear
- **AI Usage**: Task tracking, note-taking, context loading

## KEY INSIGHTS FOR AI ASSISTANT OPTIMIZATION

### Critical Tools to Support
1. **git** - Absolute essential, all projects
2. **pytest** - Python testing, majority of projects
3. **black/ruff** - Python formatting, standard practice
4. **dotnet** - C# projects, significant usage
5. **Documentation tools** - Markdown editing is VERY common

### Workflow Patterns
- **Test-Driven**: Run tests frequently after code changes
- **Format-on-Save**: Auto-format before commits
- **Documentation-Heavy**: README, CLAUDE.md updated constantly
- **Script-Driven**: Shell scripts for common tasks
- **Issue-Tracked**: GitHub issues integrated into commits

### IdlerGear Adoption Evidence
- **6/9 projects** have IdlerGear rules in `.claude/`
- Pattern: `/start` command likely runs `idlergear context`
- **High value**: AI assistant knowledge management is a solved problem in this codebase

### Recommended Tool Priority for Goose/Claude Code
1. **git** (commit, status, diff, log) - 100% priority
2. **pytest** - 67% of projects, high-value
3. **black/ruff** - 67% of projects, quality enforcement
4. **dotnet** - 33% of projects, but critical for those
5. **grep/ripgrep** - Code search (inferred from usage)
6. **find** - File discovery (inferred from usage)
7. **read/write files** - Obvious requirement
8. **idlergear** - Knowledge management integration

### Goose-Specific Recommendations
1. **Auto-format on edit**: Run black/ruff automatically
2. **Auto-test on save**: Run pytest in background
3. **Smart git commit**: Generate conventional commit messages
4. **Documentation sync**: Auto-update docs when code changes
5. **IdlerGear integration**: Load context on session start
